# 链接
## 1.概念
#### 链接：
是将各种代码和数据片段收集并组合成为一个单一文件的过程，
 这个文件可被加载（复制）到内存并执行。 

#### 链接可在不同阶段进行
  静态链接：编译时
  动态链接：加载时、运行时

#### 链接的作用
  加速编译，使得分离编译 (separate compilation) 成为可能
  多人协作，可以把一个大型的应用程序分解为更小、更好管理的模块，可以独立地修改和编译这些模块

## 2.符号解析与重定位

### 2.1符号


#### 链接器符号

| 符号类型 | 定义 | 示例 | 备注 |
|---------|------|------|------|
| 全局符号 | 由模块 m 定义并能被其他模块引用的符号 | 非静态的 C 函数和全局变量 | "全局"在此处指所有模块的集合 |
| 外部符号 | 由其他模块定义并被模块 m 引用的全局符号 | 在其他模块中定义的非静态 C 函数和全局变量（如在文件开头使用 `extern` 声明的变量） | - |
| 局部符号 | 只被模块 m 定义和引用的符号 | 带 `static` 属性的 C 函数和全局变量<br>局部链接器符号和局部变量不同 | "局部"在此处指本模块，而不是一个函数的内部 |

![alt text](360截图20251208172903555.jpg)

#### 符号表
可以把**符号表**直观地理解成——程序的“通讯录 / 索引表”
假设你的程序是一本书：
函数名、全局变量名、标签名 → 就像书里的“章节名、人物名”
符号表 → 就是这本书后面的索引页

索引页告诉你：
这个名字 叫什么
它在书的 哪一页（地址）
占了 多少页（大小）
它是 章节（函数）还是插图（变量）
是 内部用的 还是 对外公开的
```c
int global;
void foo() {}
```
| 名字     | 地址        | 大小 | 类型 | 作用范围 |
| ------ | --------- | -- | -- | ---- |
| global | 0x804a000 | 4  | 数据 | 全局   |
| foo    | 0x80483f0 | …  | 函数 | 全局   |

### 2.2符号解析
**符号解析**——将每个**引用**  与输入的可重定位目标文件的符号表中的一个**符号定义**关联起来

局部符号
1. 在相同模块中定义和引用
2. 编译器只允许每个模块中每个局部符号有一个定义

全局符号(3种情况)
1. 符号在当前模块中无定义：
 编译器会假设该符号是在其他某个模块中定义的，生成一个链接器符号表条目，并把它交给链接器处理
 2. 如果链接器在它的任何输入模块中都找不到这个被引用符号的定义：
 输出一条错误信息并终止
 3. 还有一种情况：多个目标文件可能会定义相同名字的全局符号(用强弱符号处理)

#### 多重定义全局符号的解析
![alt text](360截图20251215175305235.jpg)
链接器符号规则:
1. 不允许有多个同名的强符号，**每个强符号只能定义一次**
 否则：链接错误

 2. 如果有一个强符号和多个弱符号同名，那么**选择强符号**
 对弱符号的引用会被解析到强符号

 3. 如果有多个弱符号同名，那么从这些弱符号中**任意选择一个**

#### 与静态库链接
符号解析的范围
1. 本模块内
2. 其它模块
3. 库函数 

##### 静态库的意义：提供符号定义供符号解析
| 项目          | 说明                           |
| ----------- | ---------------------------- |
| 符号解析的输入     | 各个 `.o` 文件中的 **符号引用 / 符号定义** |
| 静态库提供什么     | **大量符号定义**（函数、全局变量）          |
| 链接器的目标      | 为每个未定义符号找到唯一的定义              |
| 静态库的作用      | 作为**“候选符号定义集合”**             |
| 是否整库加载      | ❌ 不会                         |
| 何时从库中取 `.o` | 当库中的某个 `.o` **能解析当前未定义符号**   |
| 取出来后        | 该 `.o` 被加入链接过程，产生新符号         |
| 解析是否继续      | ✅ 新符号可能触发进一步解析               |
| 最终结果        | 所有符号都被解析，或报错                 |

##### 静态库的优势
| 项目       | 直观解释                                         |
| -------- | -------------------------------------------- |
| 静态库作用    | 把多个相关的函数代码**提前打包**，供程序链接使用                   |
| 解决的问题    | 避免手动链接大量 `.o` 文件，减少依赖和出错                     |
| 组成方式     | 多个**可重定位目标文件（`.o`）**的集合                      |
| 文件本质     | **archive（归档）文件**                            |
| 文件后缀     | `.a`                                         |
| 内部结构     | 有头部，记录每个 `.o` 的**大小和位置**                     |
| 使用方式     | 编译/链接时在命令行指定库文件名                             |
| 示例       | `gcc main.c /usr/lib/libm.a /usr/lib/libc.a` |
| 链接时行为    | **只复制程序实际用到的目标模块**                           |
| 对可执行文件大小 | 比“整库拷贝”小，按需包含                                |
| 对程序员的好处  | 只需记住**少量库名**，不用关心内部 `.o`                     |
| 运行时依赖    | ❌ 无依赖（代码已拷进可执行文件）                            |
| 适用系统     | Linux / Unix                                 |
| 与动态库区别   | 静态库在**链接时**拷贝代码，动态库在**运行时**加载                |

#####   命令行上的库和目标文件的顺序：定义在引用后
#####   关键
| 情况          | 为什么不用 / 必须重复             |
| ----------- | ------------------------ |
| A：`.o ↔ .a` | `.o` 的符号**一次性全部加载**，永远可见 |
| B：`.a ↔ .a` | `.a` 是**按需、一次性扫描**，不会回头  |

所以.o目标文件加载一次即可，不用重复加载
.a静态库需要反复加载，因为一次只能加载前面依赖的.o文件

#### 程序头部表
**程序头部表**：运行时装载说明书，告诉操作系统“这程序该怎么放进内存”。
| 字段         | 直观含义                  |
| ---------- | --------------------- |
| `p_type`   | 这段是干嘛的（代码？数据？动态链接？）   |
| `p_offset` | 在 ELF 文件里的位置          |
| `p_vaddr`  | 放到内存的虚拟地址             |
| `p_filesz` | 文件中有多大                |
| `p_memsz`  | 内存中要占多大（如 `.bss` 会变大） |
| `p_flags`  | 权限（读 / 写 / 执行）        |
| `p_align`  | 对齐要求                  |
##### 什么时候使用
| 场景      | 是否用  |
| ------- | ---- |
| 运行程序    | ✅ 必须 |
| 动态链接    | ✅ 必须 |
| 链接静态库   | ❌ 不用 |
| 编译 `.o` | ❌ 没有 |
